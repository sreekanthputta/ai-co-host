## 3. Riskiest Technical Part and How You De-Risk It

The riskiest part is **not Moss**. It is the full real-time voice loop:

> microphone audio → transcription → chunking → Moss retrieval → Minimax classification → whisper generation → TTS/audio playback → LiveKit delivery

Your claimed **400ms** is dangerous. In a hackathon setting, that number will make you look delusional if the demo lands in 1.8 seconds.

You can still win with 1–2 seconds if the UX feels useful.

### Primary risk: latency and false positives

If the whisper arrives too late or says something dumb, the demo dies.

### De-risking plan

#### A. Do not whisper on every sentence

Use a lightweight trigger gate before calling Minimax.

Examples:

- sentence contains “moat,” “competitor,” “Clay,” “pricing,” “proprietary,” “exclusive,” “nobody else,” “we don’t compete,” etc.
- or embedding similarity to high-risk context crosses a threshold.

Moss can retrieve constantly, but Minimax should only judge on likely risky moments.

#### B. Use Moss retrieval to narrow context brutally

Do not ask Minimax, “Did anything bad happen?”

Ask:

> “Current utterance: X. Retrieved memories: A, B, C. Is there a contradiction or competitive landmine? If yes, produce a 12-word whisper. If no, return NONE.”

This keeps Minimax fast and bounded.

#### C. Put strict output constraints on Minimax

Minimax response schema:

```json
{
  "alert": true,
  "type": "contradiction | competitor_landmine",
  "confidence": 0.0,
  "whisper": "12 words max"
}
```

Only whisper if confidence > 0.75.

#### D. Have a visual fallback

If TTS or earpiece audio flakes, the dashboard still shows:

- transcript,
- Moss match,
- Minimax reasoning,
- generated whisper.

That means the demo can survive audio failure.

#### E. Use headphones, not a cute invisible earpiece

Do not burn time on earpiece theater. Use normal headphones. Say “earpiece” in the product story; demo with headphones.

#### F. Pre-ingest the demo corpus

Do not scrape competitor sites live. Do not parse decks live. Preload the context before demo.

You can show an “upload deck / add competitor” button if you want, but don’t depend on it.

If Moss is removed, the product loses real-time semantic memory. Minimax would need the whole transcript and corpus every time, which is slower, more expensive, and less reliable.

**Visible break:** the dashboard can no longer surface the relevant prior claim instantly.

### Minimax

Moss can retrieve candidates, but it cannot decide:

- contradiction vs refinement,
- landmine vs harmless mention,
- whether to whisper,
- how to phrase the whisper tactfully.

If Minimax is removed, you get dumb alerts:

> “Moat mentioned.”

or brittle rules:

> “proprietary data != open source.”

That’s not the product. The product is a nuanced live coach.

**Visible break:** without Minimax, the whisper becomes noisy, robotic, or wrong.

### LiveKit

LiveKit must be more than “we streamed some audio.”

Make it visibly power the live call and whisper channel:

- investor and founder are in a LiveKit room,
- transcript is tied to live participant audio,
- coach whisper is sent privately to the founder, not broadcast to the investor,

If LiveKit is removed, you no longer have the “in-call private earpiece co-pilot.” You have a post-call analysis tool or a chatbot beside Zoom.

## Recommended 20-Hour Build Scope

### Build only this

1. LiveKit room with two participants:
   - founder,
   - investor/customer.

2. Realtime-ish transcript:
   - use whatever ASR path is fastest to integrate.
   - if needed, browser speech recognition is acceptable for demo backup, but route the call through LiveKit.

3. Moss memory store:
   - preload deck claims and competitor battlecards,
   - insert live transcript chunks as they happen,
   - query Moss for top-k relevant memories on each finalized sentence.

4. Minimax judge:
   - receives current sentence + top Moss matches,
   - outputs `NONE` or short whisper.

5. Private whisper:
   - send whisper to founder via LiveKit data/audio channel.
   - display on dashboard.

6. Dashboard:
   - live transcript,
   - Moss retrieval result,
   - Minimax decision,
   - whisper sent.

### Do not build

- full CRM integration,
- real earpiece hardware,
- broad buying-signal detection,
- web scraping,
- investor public-statement ingestion,
- multi-agent debate,
- automatic deck parsing unless trivial,
- post-call analytics.

# ARCHITECTURE

**Moss (essential).** Stores three categories of memory: prior claims (from pitch deck), competitor facts (battlecards), and personal coaching notes ("avoid saying X, say Y instead"). On every finalized utterance, Moss returns top-3 semantically relevant memories. <10ms matters because the conversational correction window — the time before the speaker moves to the next sentence — is roughly 800ms. Total budget: retrieval + LLM judgment + TTS delivery. A 200ms retrieval eats 25% of the budget and pushes whispers past the window. We prove this live with the latency toggle. Moss must surface paraphrased semantic matches (e.g., "proprietary data" conflicting with "open-source community moat"), which kills the "this is just a hashmap" objection.

**Minimax (essential).** Receives the current utterance and Moss's top-3 candidates. Returns strict JSON: `{alert, type, confidence, whisper}`. It decides whether the moment is *actually* a contradiction vs. a refinement, whether the whisper is worth interrupting for, and how to phrase it in 12 words tactfully. Without Minimax: retrieval spam ("Found related memory: moat") instead of coaching. Minimax's speed/quality combo earns its place — slower frontier models break the latency budget; weaker fast models hallucinate contradictions and erode trust.
