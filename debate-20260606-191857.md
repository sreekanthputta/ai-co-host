## 3. Riskiest Technical Part and How I’d De-Risk It

### Risk A: Real PSTN call interception

This is the biggest trap. Real mobile call monitoring/interruption is not something you casually ship in 20 hours. iOS/Android do not let random apps silently listen to live cellular calls. PSTN bridging via Twilio/SIP plus LiveKit might be possible, but it’s integration hell under time pressure.

**De-risk: do not do PSTN.**

Build a **LiveKit-native “safe calling” room**. The grandma side is a browser/mobile web page that looks like a phone call. The scammer side is another browser tab or phone. Sentinel is a LiveKit agent.

Failure mode if LiveKit/PSTN breaks live: if you attempted real phones, you die on stage. If you use a LiveKit room, worst case you reload the browser and keep going.

### Risk B: Corpus quality

Your “10,000+ FTC/AARP/Reddit transcripts” claim is dangerous. FTC/FCC/NCOA have great scam descriptions, but not necessarily thousands of clean call transcripts. Reddit has examples, but scraping Reddit in 20 hours can get annoying and legally/API-wise messy.

**De-risk: build a high-quality 300–800 snippet corpus, not a fake 10k corpus.**

Use:
- FTC scam alerts
- FCC grandparent scam warnings
- NCOA elder scam pages
- state AG scam examples
- AARP fraud watch articles if accessible
- manually written paraphrases for each tactic category

Moss index should contain **semantic tactic cards**, not raw full documents:

```json
{
  "tactic": "family emergency secrecy pressure",
  "examples": [
    "please don't tell mom and dad",
    "don't call anyone else, I’m embarrassed",
    "keep this between us",
    "if anyone finds out, I’ll be in more trouble"
  ],
  "recommended_intervention": "Ask the caller to hang up and call the family member directly using a saved number."
}
```

This makes Moss matter because the judge can paraphrase:

> “Let’s not involve your daughter yet.”

A regex may miss that. Moss should still retrieve the “secrecy pressure” tactic.

### Risk C: Latency chain

Moss might be <10ms, but your actual chain is:

audio → STT partial transcript → utterance chunking → Moss search → Minimax classification → TTS/interruption

That is not 10ms. Don’t overclaim “RAG would be 800ms too late.” Judges may smell bullshit.

**De-risk: frame Moss as enabling per-utterance semantic scanning without making latency explode.**

Say:

> “The agent can afford to run semantic retrieval on every partial utterance because Moss search is fast enough to be in the live audio loop.”

That’s credible.
